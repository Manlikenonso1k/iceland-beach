## 📌 What Was Built

A complete end-to-end beach event ticketing system layered on top of the existing Iceland Beach Resort PHP site. It has two surfaces:

| Surface | Stack | Role |
|---|---|---|
| **Public form** | PHP Legacy (`ticket.php`) | Customer purchase form + payment |
| **Admin dashboard** | Laravel 12 + Filament 3 (`laravel-app/`) | Order management, QR check-in, receipts |

Customers are **guests — no login required**. They buy a ticket, pay via Paystack or TGI Titan, receive a receipt email with a QR code, and present the QR at the gate. Admin scans the QR to check them in.

## 🔀 Payment Flow Diagrams

### Paystack Flow
```
Customer fills ticket.php
        ↓
JS intercepts submit → opens Paystack inline popup
        ↓ (customer pays in popup)
Paystack JS callback → populates hidden #paystack_ref
        ↓
Form POSTs to POST /verify-payment
        ↓
PaymentController::paystackVerify()
  → Http::withToken(secret_key)->get('/transaction/verify/{ref}')
  → checks data.status === "success"
  → creates Ticket record (paid=true)
  → generateQr($ticket)
  → sendReceiptEmail($ticket)
        ↓
redirect → /ticket/{uuid}/success
```

### TGI Titan Flow
```
Customer fills ticket.php, selects Titan
        ↓
Form POSTs to POST /initiate-titan
        ↓
PaymentController::initiateTitan()
  → creates PENDING Ticket record (paid=false) ← save BEFORE redirect
  → Http::withHeaders(['integration-key' => api_key])
       ->post('/payment/initiate', payload)
  → receives payment URL from Titan
        ↓
redirect()->away($paymentUrl)  ← customer goes to Titan hosted page
        ↓ (customer pays on Titan)
Titan redirects back → GET /titan/callback?ref=...&status=...
        ↓
PaymentController::titanCallback()
  → normalises status (success/failed/processing)
  → finds Ticket by order_ref
  → updates paid=true, payment_ref=ref
  → generateQr($ticket)
  → sendReceiptEmail($ticket)
        ↓
redirect → /ticket/{uuid}/success
```

### Gate Entry / QR Scan Flow
```
Admin scans QR code at gate
        ↓
GET /ticket/verify/{uuid}
        ↓
PaymentController::verifyEntry()
  → if !paid      → green=false, show RED screen "INVALID — Not Paid"
  → if checked_in → show ORANGE screen "ALREADY CHECKED IN"
  → else          → update checked_in=true, show GREEN screen "VALID — Welcome!"
```

## 🛣️ Routes Reference

| Method | URI | Controller Method | Notes |
|---|---|---|---|
| `POST` | `/verify-payment` | `paystackVerify` | Paystack server-side verify |
| `POST` | `/initiate-titan` | `initiateTitan` | Titan payment initiation |
| `GET` | `/titan/callback` | `titanCallback` | Titan redirect after payment — **CSRF exempt** |
| `GET` | `/ticket/{uuid}/success` | `success` | Receipt page with QR |
| `GET` | `/ticket/verify/{uuid}` | `verifyEntry` | Gate scan — green/red screen |
| `GET` | `/admin/tickets` | Filament `ListTickets` | Admin dashboard |
| `GET` | `/admin/tickets/{record}` | Filament `ViewTicket` | Admin ticket detail |

> **CSRF Note**: Titan uses a GET redirect callback, not a POST webhook. CSRF is excluded via `->withoutMiddleware([VerifyCsrfToken::class])` directly on the route — **not** in the `$except` array. This follows the pattern in `PAYMENT_KNOWLEDGE.md`.

## 🔑 Payment Gateway Details

### Paystack
- **Auth**: Bearer token (`Authorization: Bearer sk_...`) via `Http::withToken()`
- **Initiation**: Inline JS popup (`PaystackPop.setup()`) — **no server redirect**
- **Amount**: Sent in **kobo** (naira × 100)
- **Verification endpoint**: `GET /transaction/verify/{reference}`
- **Success check**: `data.status === "success"` in response JSON

### TGI Titan (TGIPAY)
- **Auth**: `integration-key` header — **no HMAC signature** (confirmed in PAYMENT_KNOWLEDGE.md)
- **Amount**: Sent as **naira float** (NOT kobo) — divide kobo by 100 before sending
- **Initiation endpoint**: `POST /payment/initiate`
- **Payload fields**: `customerFirstName`, `customerLastName`, `customerEmail`, `amount`, `transactionReference`, `currency`
- **Payment URL**: Returned at `body.data.url` in response
- **Callback**: GET redirect with `?ref=...&status=...` query params
- **Status normalisation**:
  ```php
  match ($status) {
      'success', 'successful', 'completed', 'paid' => 'success',
      'failed', 'cancelled', 'canceled'             => 'failed',
      default                                        => 'processing',
  }
  ```

## 🖼️ QR Code System

- **Library**: `simplesoftwareio/simple-qrcode` v4.2.0
- **Format**: SVG (scales perfectly at any size)
- **Error correction**: `H` (highest — survives partial damage)
- **Size**: 300px
- **Storage**: `storage/app/public/qrcodes/{uuid}.svg`
- **Public URL**: `https://icelandbeach.com/laravel-app/public/storage/qrcodes/{uuid}.svg`
- **QR content**: The full URL of `/ticket/verify/{uuid}` — admin scans this at gate
- **Display on success page**: Inline SVG via `file_get_contents()` — works fully offline after first load
- **Display in email**: Base64-encoded SVG data URI — no external image server needed

> ⚠️ QR is saved **once** when payment is confirmed. Never regenerate on every page load.

## 🚧 Known Gotchas & Lessons

| Issue | Detail |
|---|---|
| `exec()` disabled on host | `storage:link` artisan command fails. Use `ln -sf` directly. |
| Titan amount is naira, not kobo | TGIPAY expects a float naira value. Convert back: `$amountKobo / 100` |
| Paystack amount IS kobo | Paystack JS and API both expect kobo (`price × qty × 100`) |
| CSRF on Titan callback | Titan uses GET redirect, not POST webhook. Excluded via `->withoutMiddleware()` on the route |
| Save ticket BEFORE Titan redirect | Create the pending record first — you need the `order_ref` to match on callback |
| Never expose `id` in URLs | Always use `uuid` in routes. The `id` column is internal only |
| QR must be saved to a file | Never regenerate QR on every page load — generate once on payment success |
| Route cache on production | After any route changes, run `php artisan route:cache` or routes won't update |
